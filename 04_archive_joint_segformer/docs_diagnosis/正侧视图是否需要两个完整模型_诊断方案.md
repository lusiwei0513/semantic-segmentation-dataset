# 正视图与侧视图是否需要两个完整模型：数据诊断与实验判定方案

## 1. 目的

本方案用于判断轨道车辆正视图和侧视图应该采用哪种建模方式：

### 方案 A：推荐默认方案

```text
一个 SegFormer-B0 共享编码器
├── 正视图分割头：车身主体、前挡风玻璃
├── 正视图关键点头：车头尖端
└── 侧视图分割头：车身主体、转向架、车门、前挡风玻璃
```

### 方案 B：两个完整模型

```text
正视图 SegFormer-B0
├── 正视图分割头
└── 车头尖端关键点头

侧视图 SegFormer-B0
└── 侧视图分割头
```

本方案的目标不是仅凭“正侧视图看起来差异很大”就拆模型，而是通过数据检查和小规模对照实验判断是否真的存在严重负迁移。

---

# 2. 必须先明确的结论

## 2.1 标签不同，不等于必须使用两个完整模型

正视图和侧视图标签不同，只说明它们需要不同的输出头。

它并不能证明底层视觉特征也必须完全分开学习。

正视图和侧视图仍共享大量特征：

- 车辆外轮廓；
- 金属表面纹理；
- 玻璃反光；
- 车身边缘；
- 光照和阴影；
- 轨道车辆外观结构。

因此，在总数据量只有约 200 张时，默认应该采用：

> 共享编码器 + 正视图专用头 + 侧视图专用头。

## 2.2 两个完整模型的真正判断依据

只有出现以下情况，才应认真考虑两个完整模型：

1. 正视图和侧视图由完全不同的设备或成像流程采集；
2. 分辨率、色彩、曝光、镜头畸变和背景存在系统性差异；
3. 共享编码器训练时出现明显负迁移；
4. 两个独立模型在严格交叉验证中稳定优于共享模型；
5. 每个视图的数据量足够支撑独立模型；
6. 两个模型在部署端本来就运行于不同设备。

---

# 3. 最重要的混淆：视图和设备是否完全绑定

假设：

```text
所有正视图都由设备 A 拍摄
所有侧视图都由设备 B 拍摄
```

那么数据中：

```text
view = front  <=> device = A
view = side   <=> device = B
```

此时“视图”和“设备”完全混淆。

即使一个分类器能以 100% 准确率区分正视图和侧视图，也无法判断它依据的是：

- 正视图与侧视图的几何差异；
- 设备 A 与设备 B 的颜色和清晰度差异；
- 两者共同作用。

因此：

> 当设备和视图一一绑定时，单靠现有数据无法严格分离设备差异和视图差异。

## 3.1 最可靠的补充数据方式

建议额外采集一个很小的交叉数据集：

```text
设备 A：拍摄正视图和侧视图
设备 B：拍摄正视图和侧视图
```

理想情况下：

- 选取 10～20 辆有代表性的车辆；
- 每辆车用两个设备分别拍正视图和侧视图；
- 保持尽可能接近的距离、光照和背景；
- 不一定需要全部精标，可以先用于成像域检查；
- 若能精标其中 20～40 张，则可以做跨设备验证。

有了交叉数据，才能分别判断：

- 固定视图时，设备差异有多大；
- 固定设备时，视图差异有多大；
- 共享编码器是否能适应两个设备。

# 4. metadata.csv 必须增加的字段

建议数据表至少包含：

```csv
sample_id,image_path,vehicle_id,view,device_id,width,height,source_batch
```

字段说明：

| 字段 | 说明 |
|---|---|
| sample_id | 每张图唯一编号 |
| image_path | 图像路径 |
| vehicle_id | 同一车辆或同一列车编号 |
| view | front 或 side |
| device_id | 拍摄设备或成像系统编号 |
| width | 原始宽度 |
| height | 原始高度 |
| source_batch | 拍摄批次、日期或来源 |

如果不知道设备型号，也应至少使用：

```text
front_camera
side_camera
unknown_01
scanner_A
scanner_B
```

不要全部留空。

# 5. 自动数据诊断内容

配套脚本 `check_front_side_domain.py` 将检查以下内容。

## 5.1 数据数量

输出：

- 总图像数；
- 正视图图像数；
- 侧视图图像数；
- 独立 vehicle_id 数量；
- 每个设备图像数；
- 每个视图对应的设备分布；
- 每个设备对应的视图分布。

## 5.2 设备与视图混淆检查

脚本生成列联表：

```text
device_id × view
```

如果某个设备只拍摄一种视图，并且每种视图只来自一个设备，报告中标记：

```text
SEVERE_CONFOUNDING
```

此时不能根据图像分类结果直接判定需要两个模型。

## 5.3 图像基础统计

逐张统计：

- 宽度；
- 高度；
- 长宽比；
- RGB 均值；
- RGB 标准差；
- 灰度均值；
- 对比度；
- 饱和度；
- 过曝比例；
- 欠曝比例；
- 拉普拉斯清晰度；
- JPEG 或文件大小信息。

这些统计分别按：

```text
view
device_id
view + device_id
```

汇总。

## 5.4 预训练特征域差异

脚本使用 ImageNet 预训练 ResNet-50 提取图像嵌入，并计算：

- 正侧视图特征中心余弦距离；
- 设备之间特征中心距离；
- 线性 MMD；
- PCA 二维分布图；
- 视图线性探针准确率；
- 设备线性探针准确率；
- 在同一视图内部预测设备的准确率；
- 在同一设备内部预测视图的准确率。

注意：

> “正侧视图探针准确率高”是正常现象，不能单独作为拆成两个模型的依据。

更有价值的是：

```text
固定同一视图后，设备仍然很容易被识别
```

这说明设备带来的成像域差异较强。

# 6. 自动诊断结果如何解释

## 6.1 情况一：只有视图差异，没有明显设备差异

典型特征：

- 两种视图由同一设备或相似设备采集；
- 分辨率和色彩统计接近；
- 设备分类探针效果接近随机；
- 正侧视图分类很容易，但这是几何差异导致；
- 共享模型没有明显负迁移。

结论：

```text
使用一个共享 SegFormer-B0 编码器
+ 正视图专用头
+ 侧视图专用头
```

## 6.2 情况二：设备差异明显，但数据量很小

典型特征：

- 正侧视图来自不同设备；
- 色彩、清晰度、分辨率差异明显；
- 设备分类探针准确率很高；
- 每种视图只有约 100 张。

结论仍然优先：

```text
共享编码器
+ 视图专用头
+ 设备相关归一化或增强
```

可尝试：

- 每个设备单独计算归一化统计；
- 使用颜色和曝光增强；
- 使用 InstanceNorm 或 GroupNorm；
- 在输入端增加 device_id；
- 使用 Device-Specific BatchNorm；
- 保持两个视图分开组成 batch；
- 分别设置正侧视图输入分辨率。

只有严格实验表明共享模型明显更差，才拆成两个完整模型。

## 6.3 情况三：设备差异明显，并且共享训练稳定负迁移

满足以下多项：

- 两个独立模型在五折测试中稳定提高；
- 每个视图的提升都不是偶然；
- 共享模型在其中一个视图明显下降；
- 多折结果标准差较小；
- 提升大于预设阈值；
- 不只是训练集指标更高；
- 部署条件允许维护两个模型。

结论：

```text
可以使用两个完整模型
```

## 6.4 情况四：设备与视图完全混淆

例如：

```text
front 全部来自设备 A
side 全部来自设备 B
```

结论：

```text
现有数据无法严格判断设备是否导致必须拆模型。
```

此时应：

1. 默认采用共享编码器 + 两个专用头；
2. 做共享与独立模型五折对照；
3. 尽量补采交叉设备数据；
4. 不把“域分类准确率高”直接解释为必须拆模型。

# 7. 最终必须做的小规模模型对照

仅靠图像统计不能最终决定模型结构。

必须比较：

## 实验 A：共享编码器

```text
SegFormer-B0 共享 MiT-B0
├── FrontSegHead
├── NoseTipHead
└── SideSegHead
```

## 实验 B：两个完整模型

```text
Front SegFormer-B0
Side SegFormer-B0
```

两组实验必须保持：

- 相同 ImageNet 预训练；
- 相同数据划分；
- 相同增强；
- 相同 epoch；
- 相同早停规则；
- 相同输入尺寸；
- 相同随机种子集合；
- 相同评价指标。

推荐使用 5 折 Group Cross Validation。

# 8. 模型选择指标

正视图：

- body IoU；
- windshield IoU；
- nose_tip 归一化误差；
- PCK@0.02；
- PCK@0.05。

侧视图：

- body IoU；
- bogie IoU；
- door IoU；
- windshield IoU。

综合分数可以定义为：

```text
front_score
= mean(body_iou, windshield_iou, pck_005)

side_score
= mean(body_iou, bogie_iou, door_iou, windshield_iou)

overall_score
= 0.5 × front_score + 0.5 × side_score
```

也可以根据老师要求调整权重。

# 9. 推荐判定阈值

以下是工程判定建议，不是普适定理。

## 保留共享编码器

满足任一情况就优先保留共享编码器：

- 两个独立模型的 overall_score 提升小于 1 个百分点；
- 提升只出现在单个 fold；
- 提升的标准差大于平均提升；
- 正视图提升但侧视图下降，或反之；
- 两个完整模型明显过拟合；
- 每个视图有效训练样本少于约 150 张；
- 部署维护成本明显上升。

## 考虑两个完整模型

只有同时满足以下大部分条件：

1. 独立模型相对共享模型平均提升至少 2～3 个百分点；
2. 5 折中至少 4 折方向一致；
3. 正视图和侧视图都没有明显退化；
4. 提升主要出现在测试集而非训练集；
5. 设备或成像差异有明确证据；
6. 每个视图数据足够，或独立模型没有明显过拟合；
7. 推理部署允许维护两个模型。

## 默认结论

对于当前总计约 200 张数据：

> 默认选择共享编码器 + 正侧视图专用头。

两个完整模型必须用严格对照实验来证明，而不是仅凭标签不同或设备不同直接决定。

# 10. 进一步检查负迁移的方法

如果共享模型效果不佳，可检查：

## 10.1 梯度冲突

分别计算正视图损失和侧视图损失对共享编码器参数的梯度，并计算余弦相似度。

```text
cosine > 0：
两个任务总体方向一致

cosine ≈ 0：
两个任务基本独立

cosine < 0：
存在梯度冲突
```

如果长期、大量 batch 中梯度余弦显著为负，可以尝试：

- PCGrad；
- GradNorm；
- 分阶段训练；
- 共享前两级编码器，后两级分支；
- 最后才考虑两个完整模型。

## 10.2 部分共享

在完全共享与完全拆分之间，还有更合理的中间方案：

```text
共享 MiT Stage 1、2
├── 正视图 Stage 3、4 + 专用头
└── 侧视图 Stage 3、4 + 专用头
```

如果底层纹理和边缘可共享，但高层结构冲突，这种方案通常比两个完整模型更节省数据和参数。

# 11. Agent 执行顺序

1. 给 metadata.csv 补充 device_id 和 vehicle_id；
2. 运行 `check_front_side_domain.py`；
3. 检查是否存在设备—视图完全混淆；
4. 查看基础统计和 PCA 图；
5. 查看设备分类探针；
6. 输出自动诊断报告；
7. 不根据单一指标直接拆模型；
8. 优先搭建共享编码器 + 视图专用头；
9. 训练阶段再增加两个完整模型作为对照；
10. 根据五折结果使用本文档阈值作最终决定。
